# Learning OS MVP 方案收敛设计稿

## 1. 文档目标

本文档用于在现有 [prd.md](/Users/zhenganlin/Desktop/04-zhen/learning-os/doc/prd.md) 基础上，收敛 Learning OS 的第一版实现范围，明确产品闭环、关键交互、数据边界和错误兜底策略。

本稿聚焦 `MVP 如何做浅而完整的闭环`，不追求一次性定义全部长期能力。

## 2. 结论摘要

Learning OS 的 MVP 不应被定义为“全自动知识库系统”，而应定义为：

`一个本地优先、AI 辅助、用户确认后入库的个人学习整理器`

MVP 的核心闭环为：

`导入内容 -> Agent 生成整理结果 -> 用户确认/微调 -> 正式入库 -> 生成今日复习 -> 用户完成复习 -> 可搜索找回`

这个闭环强调：

- 全链路都要有，但每个环节先做浅版
- AI 负责生成候选结果，不直接决定最终知识库内容
- 正式知识库只接收用户确认后的内容
- 第一版优先验证“是否可持续使用”，而不是“是否足够智能”

## 3. MVP 产品定位调整

### 3.1 从“自动沉淀系统”调整为“半自动整理器”

原 PRD 已经覆盖导入、知识抽取、复习、搜索、推荐等能力，但第一版如果同时深做这些模块，会把工程复杂度抬得过高。

因此建议将产品定位调整为：

- 系统自动生成知识候选结果
- 用户进行轻量审核与修正
- 用户确认后再写入正式知识库

这会让第一版更容易成立，因为它允许模型输出“有帮助但不完美”。

### 3.2 MVP 成功标准

MVP 成功不以“AI 输出是否完美”为标准，而以以下用户任务是否完成为标准：

1. 用户能导入一篇 URL、Markdown 或文本内容
2. 系统能生成一组可审核的核心知识点与候选卡片
3. 用户能完成一次确认并将结果入库
4. 用户能在当天完成一次复习
5. 用户能在之后通过搜索找回这些知识

## 4. 关键设计原则

### 4.1 全链路浅覆盖

第一版必须从导入跑到复习结束，但每一段都应采用浅实现，不做重工程优化。

### 4.2 分层输出

导入结果采用分层结构：

- `核心知识点`：3-5 个，默认选中
- `候选知识点`：0-10 个，默认不选
- `候选卡片`：挂在知识点下方，允许逐张保留或删除

### 4.3 确认优先于创作

整理结果页的目标是快速审核，不是深度知识编辑。第一版优先支持：

- 删除低质量结果
- 修改明显错误
- 确认保留项

不优先支持：

- 复杂知识关系编辑
- 大段笔记重写
- 批量知识结构编排

### 4.4 证据可追溯

每个核心知识点应尽可能附带对应证据片段，帮助用户判断 AI 是否抽取得当。

## 5. 核心业务闭环

### 5.1 主流程

```text
用户发起导入
  -> 保存原始 Source
  -> 创建 IngestionSession
  -> 创建 AgentTask
  -> Agent 生成候选结果
  -> 用户进入整理结果确认页
  -> 用户确认部分或全部候选结果
  -> 系统正式入库 Concept / Note / ReviewCard
  -> 已入库卡片进入复习队列
  -> 用户完成今日复习
  -> 系统记录 ReviewLog 与掌握度
```

### 5.2 关键状态拆分

建议将状态拆成两层：

`AgentTask 状态`

```text
pending -> running -> succeeded / failed
```

只表示 AI 整理任务执行情况。

`IngestionSession 状态`

```text
created -> processing -> reviewable -> confirmed -> imported / failed / discarded
```

表示一次导入业务流程的真实进展。

这样可以清楚区分：

- Agent 成功：模型产出了结果
- 入库成功：用户确认并写入正式知识库

## 6. 整理结果确认页设计

整理结果确认页是 MVP 的质量闸门，建议作为正式主流程节点存在。

### 6.1 页面结构

页面建议包含四个区域：

1. 来源摘要区
2. 核心知识点区
3. 候选知识点区
4. 候选卡片预览区

### 6.2 来源摘要区

展示：

- 来源标题
- 导入类型
- 导入时间
- 原文摘要
- 领域提示

目标是让用户快速确认这次整理的是哪篇内容。

### 6.3 核心知识点区

默认展示 3-5 个核心知识点，默认勾选。

每个条目建议展示：

- 标题
- 一句话摘要
- 领域 / 主题
- 证据片段
- 预计生成卡片数

允许用户操作：

- 取消勾选
- 修改标题
- 修改一句话摘要
- 修改领域 / 主题
- 查看证据片段

### 6.4 候选知识点区

默认折叠，默认不选中。

这些内容用于补充潜在价值，但不应让用户感觉必须全部处理。

### 6.5 候选卡片区

卡片建议按知识点组织，而不是按整篇内容平铺。

每张卡片至少支持：

- 保留
- 删除
- 轻改问题文本
- 查看答案与解释

### 6.6 第一版刻意不做的交互

为控制复杂度，第一版不建议在此页支持：

- 知识点拖拽重排
- 知识关系人工编排
- 笔记全文深度编辑
- 大量批量操作

## 7. MVP 范围重排

### 7.1 必须保留

- URL / Markdown / 文本导入
- Agent 生成核心知识点、候选知识点、候选卡片
- 整理结果确认页
- 用户确认后正式入库
- 知识点列表与详情页
- 今日复习
- ReviewLog 记录
- 基础掌握度更新
- 关键词搜索
- 本地数据目录初始化
- 日志与失败重试

### 7.2 建议降级实现

#### 重复知识点判断

从“自动合并”降级为“疑似重复提醒 + 用户决定是否继续入库”。

#### 语义搜索

从 MVP 硬依赖降级为增强项。可以预留接口和存储层，但不作为第一版是否可用的核心判断标准。

#### 推荐系统

从推荐引擎降级为规则展示：

- 今日到期复习
- 最近导入内容
- 低掌握度知识点

#### 知识关系

从全量关系自动生成降级为：

- 仅保留扩展位
- 或只支持 `related`
- 或第一版不展示关系

#### 掌握度

从复杂综合打分降级为简单可解释模型。

### 7.3 建议移出 MVP

- 自动知识点合并
- 复杂 Topic 树
- 全量卡片类型
- 高级推荐公式
- 重型学习统计面板
- 多格式导出体系
- 复杂 PDF 解析
- 知识图谱可视化

### 7.4 卡片类型建议

第一版建议只保留：

- `qa`
- `cloze`

其他类型后置。

## 8. 数据模型补充建议

现有 PRD 已有 `Source`、`Concept`、`Note`、`ReviewCard`、`ReviewLog`、`AgentTask` 等正式实体，但缺少“导入候选层”。

### 8.1 IngestionSession

表示一次完整导入业务流程。

建议字段方向：

- `id`
- `sourceId`
- `latestAgentTaskId`
- `status`
- `domainHint`
- `tags`
- `createdAt`
- `updatedAt`
- `confirmedAt`
- `importedAt`

### 8.2 ConceptCandidate

表示确认页中的候选知识点。

建议字段方向：

- `id`
- `sessionId`
- `title`
- `summary`
- `domain`
- `topic`
- `tags`
- `evidenceChunkIds`
- `isCore`
- `isSelected`
- `possibleDuplicateConceptId`
- `editStatus`

### 8.3 CardCandidate

表示候选复习卡片。

建议字段方向：

- `id`
- `sessionId`
- `conceptCandidateId`
- `type`
- `question`
- `answer`
- `explanation`
- `isSelected`
- `editedByUser`

### 8.4 ImportDecision

用于记录用户如何处理候选结果。

建议记录：

- 保留
- 删除
- 修改后保留
- 标记为疑似重复
- 放弃整次入库

这层数据对后续优化 Agent 很重要。

## 9. 搜索、复习与知识库边界

### 9.1 知识库边界

只有用户确认并成功入库后的内容，才进入正式知识库。

未确认候选内容默认不应出现在：

- 知识点列表
- 全局搜索结果
- 今日复习

### 9.2 搜索边界

第一版以关键词搜索为核心。

要求：

- 已入库知识点可被搜索
- 已入库笔记可被搜索
- 搜索结果可跳回知识点详情与来源材料

### 9.3 复习边界

只有正式入库后的卡片才进入复习系统。

复习过程必须可追踪：

- 本次评分
- 上次应复习时间
- 下次应复习时间
- 用时

## 10. 错误处理与人工兜底

第一版必须承认“AI 结果不稳定”是正常情况，而不是异常情况。

### 10.1 失败类型

建议将失败分为四类：

1. 抓取失败
2. 模型处理失败
3. 结果质量不满意
4. 入库失败

### 10.2 用户可见兜底动作

#### 抓取失败

- 重新抓取
- 改用文本导入

#### 模型处理失败

- 重试整理
- 切换模型后重试
- 仅保存原文，稍后再整理

#### 结果质量不满意

- 仅保留部分核心知识点
- 删除低质量候选项
- 重新生成
- 放弃整次入库

#### 入库失败

- 重试入库
- 保留候选草稿
- 稍后继续

### 10.3 人工兜底原则

任何 AI 自动生成的核心产物，在入库前都必须允许用户：

- 删除
- 跳过
- 轻修改

这条原则是 MVP 可用性的关键。

## 11. 幂等性与恢复能力

本地桌面应用天然会遇到中断、重试和重复点击，因此需要在 PRD 中显式补充以下规则：

- 同一个 `IngestionSession` 只能成功入库一次
- 入库确认动作必须幂等
- Agent 输出要与 session/version 绑定
- 重试整理不能覆盖用户已经编辑过的候选结果
- 应用重启后仍能恢复未完成的导入会话

## 12. 验收标准修订建议

当前 PRD 中不少验收项仍停留在“功能存在”。建议改为“用户可完成真实闭环”。

### 12.1 导入中心

建议验收补充为：

- 用户导入后可看到处理中状态
- 整理完成后可进入确认页
- 用户可仅采纳部分结果
- 放弃入库时原文仍保留

### 12.2 Agent 自动整理

建议验收补充为：

- 至少生成 3 个可审核的核心知识点候选，或明确提示无法提取
- 核心知识点附带证据片段
- 卡片候选可单独保留或删除
- 整理失败时可重试且不丢失原始 source

### 12.3 知识库

建议验收补充为：

- 只有确认入库后的知识点进入知识库
- 知识点详情可追溯来源
- 用户可区分 AI 生成内容与人工修改内容

### 12.4 复习系统

建议验收补充为：

- 仅已入库卡片进入复习队列
- 复习结果会更新下次复习时间
- 次日重新打开应用仍可看到待复习内容

### 12.5 搜索

建议验收补充为：

- 已入库知识点与笔记可被关键词命中
- 搜索结果能跳转详情与来源
- 未确认候选内容不进入全局搜索

## 13. 分阶段落地建议

如果后续继续拆实现计划，建议按以下顺序推进：

1. Electron + React + NestJS + 本地目录初始化
2. Source / IngestionSession / AgentTask 主流程跑通
3. 整理结果确认页与 candidate 层落地
4. 正式入库 Concept / Note / ReviewCard
5. 今日复习与 ReviewLog
6. 关键词搜索
7. 降级版推荐与轻量统计

## 14. 本次收敛后的 MVP 定义

最终推荐的 MVP 定义如下：

`用户把一篇文章、Markdown 或文本导入系统后，系统能够生成一组可审核的核心知识点和候选卡片；用户确认后将部分内容入库；入库内容可参与今日复习，并能通过搜索再次找回。`

这是第一版最小但完整、且最有机会形成日常使用习惯的产品闭环。
